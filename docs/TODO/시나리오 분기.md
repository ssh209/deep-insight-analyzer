## 현재 vs 목표

### 현재 (2-시나리오)

```
Strategist -> 타임라인 1개 -> Mitigated Forecaster -> NVI 예측 1개
```

최종 리포트에는 **baseline(무대응) vs mitigated(전략적용)** 2개 시나리오만 존재.

### 목표 (N-시나리오)

```
Strategist -> 타임라인 N개 -> Forecaster x N -> NVI 예측 N개
```

경영진에게 **3~4개 대응 시나리오**를 선택지로 제시:

| 시나리오 | 설명 | NVI 최저 | 회복 시간 | 비용 |
|:---|:---|:---:|:---:|:---:|
| A. 즉시 전면 사과 + 리콜 | CEO 직접 사과 + 전 제품 리콜 | 0.48 | 96h | 높음 |
| B. 단계적 해명 + 제한 리콜 | 해명 -> 조사 -> 해당 모델만 리콜 | 0.38 | 168h | 중간 |
| C. 무대응 (Do Nothing) | 기존 baseline | 0.25 | 336h+ | 없음 |

## 핵심 설계 결정

### 옵션 1: LangGraph `Send()` API (동적 fan-out)

```python
# Strategist가 N개 시나리오를 생성하면,
# 각각에 대해 ForecasterAgent를 동적으로 생성하여 병렬 실행
def route_scenarios(state):
    scenarios = state["scenarios"]
    return [
        Send("scenario_forecaster", {
            **state,
            "scenario_id": s["scenario_id"],
            "strategist_timeline": s["events"],
        })
        for s in scenarios
    ]

workflow.add_conditional_edges("strategist", route_scenarios)
```

**장점**: LangGraph 네이티브, 동적 병렬화, 각 시나리오 독립 실행
**단점**: `Send()` API 학습 곡선, state 관리 복잡

### 옵션 2: Strategist 내부 루프 (단순)

```python
# Strategist가 N개 타임라인을 생성한 뒤,
# 내부에서 ForecasterAgent를 직접 N번 호출
class StrategistAgent:
    def run(self, state):
        scenarios = self._generate_scenarios(state)  # N개 타임라인
        for scenario in scenarios:
            forecaster = ForecasterAgent(mode="mitigated")
            result = forecaster.run({**state, "strategist_timeline": scenario["events"]})
            scenario["forecast"] = result
        return {"scenarios": scenarios}
```

**장점**: 구현 단순, 기존 구조 유지
**단점**: 순차 실행 (병렬화 불가), Strategist 책임 과부하

### 옵션 3: ScenarioOrchestrator 별도 노드

```python
# Strategist -> ScenarioOrchestrator -> Compiler
# ScenarioOrchestrator가 N개 시나리오를 받아 각각 Forecaster 호출
class ScenarioOrchestratorAgent:
    def run(self, state):
        scenarios = state["scenarios"]
        forecaster = ForecasterAgent(mode="mitigated")
        for s in scenarios:
            result = forecaster.run({**state, "strategist_timeline": s["events"]})
            s["nvi_forecast"] = result
            s["predicted_nvi_bottom"] = min(
                result["point"] if isinstance(result, dict) else result
            )
        return {"scenarios": scenarios}
```

**장점**: 관심사 분리, Strategist 부담 없음
**단점**: 노드 1개 추가

> **추천: 옵션 3** (ScenarioOrchestrator)
> - 초기에는 순차 실행, 추후 `Send()`로 병렬화 가능
> - Strategist/Forecaster 기존 코드 변경 최소

## Pydantic 스키마

```python
class Scenario(BaseModel):
    scenario_id: str = Field(description="시나리오 식별자 (A, B, C, ...)")
    name: str = Field(description="시나리오명")
    description: str = Field(description="시나리오 설명")
    events: List[CrisisEvent] = Field(description="대응 타임라인")
    cost_estimate: str = Field(description="비용 추정 (없음/낮음/중간/높음)")
    risk: str = Field(description="시나리오 고유 리스크")
    recommendation: str = Field(description="추천 수준 (최선/차선/비권장)")

class ScenariosOutput(BaseModel):
    scenarios: List[Scenario] = Field(description="대응 시나리오 목록 (3~4개)")
```

## 파이프라인 변경

### 현재

```
Planner -> Strategist -> Mitigated Forecaster -> Analyst -> Compiler
```

### 변경

```
Planner -> Strategist -> ScenarioOrchestrator -> Analyst -> Compiler
                              |
                     (내부: Forecaster x N 호출)
```

- `Mitigated Forecaster` 노드 제거
- `ScenarioOrchestrator` 노드 추가 (Strategist 뒤)
- `strategist_timeline` (기존)은 `scenarios[0].events`로 호환
- baseline은 `scenarios[-1]` (무대응 시나리오)

---

## 하위 호환성

| 기존 필드 | 호환 방법 |
|:---|:---|
| `strategist_timeline` | `scenarios[0].events` (최선 시나리오) |
| `nvi_mitigated_forecast` | `scenarios[0].nvi_forecast` |
| `nvi_baseline_forecast` | 기존 유지 (변경 없음) |
| `defense_effect` | `scenarios[0].predicted_nvi_bottom - baseline_min` |
